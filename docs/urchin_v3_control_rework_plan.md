# Urchin v3 Control Rework Plan

**Status:** Planning document for future implementation  
**Date:** 2026-04-23  
**Scope:** Rework `urchin_v3` from �scripted seed + broad RL refinement� into a controllable, character-oriented locomotion stack built around a structured rolling engine, imitation learning, corrective dataset expansion, and narrowly-scoped residual RL.

---

## 1. Purpose

This document defines the intended control and training redesign for the `urchin_v3` robot.

The current direction has been:

1. build a programmatic controller,
2. record it as a pre-seed,
3. run RL to improve it.

That direction found a real locomotion mechanism: a spherical-harmonic-driven shape field that produces forward rolling by pushing on the bottom-rear region and reaching on the top-front region, creating a tank-tread-like rolling effect.

The rework proposed here keeps that locomotion story, but changes the architecture around it.

The new goal is **not** to discover locomotion from scratch with PPO. The new goal is to build an **authorable rolling character controller** whose locomotion is fundamentally understandable, steerable, and stylizable, with learning used mainly for compression, stabilization, and robustness.

This robot is not being optimized purely for practical utility. It is being developed as an art object / character experiment. That means controllability, expressive motion, and reliable style variation matter more than end-to-end reward maximization.

---

## 2. Core Decision

### Old framing
- Scripted locomotion is a seed.
- RL is the main optimizer.
- The learned policy is expected to �hone� the behavior into something better.

### New framing
- The scripted locomotion engine is the **main controller substrate**.
- Learning is used to:
  - imitate and compress the locomotion family,
  - stabilize timing and heading,
  - recover from off-nominal states,
  - improve robustness under randomization,
  - preserve controllability while expanding capability.

### Practical interpretation
The robot should behave as if it has:
- a **rolling engine**,
- a small number of **artist-facing controls**,
- a **bounded learned stabilizer** on top.

If the learned stabilizer is disabled, the robot should still exhibit recognizable rolling behavior.

---

## 3. Design Principles

1. **Protect the locomotion prior.**  
   The contact-dipole rolling mechanism is not incidental. It is the main discovery and should be treated as a load-bearing prior.

2. **Separate authored motion from learned correction.**  
   The base rolling behavior should be interpretable and directly steerable.

3. **Train for controllability first, optimality second.**  
   Heading control, start/stop behavior, arc turns, and recovery are more important than maximizing one scalar reward.

4. **Use RL only where it is actually strong.**  
   RL should handle robustness, timing refinement, and closed-loop compensation, not primary gait invention.

5. **Expose style as a first-class control surface.**  
   The robot should support multiple locomotion �personalities� without retraining the full stack from scratch each time.

6. **Represent phase explicitly.**  
   This gait is cyclic and contact-timed. Rolling phase should be a modeled variable, not merely something the policy must infer implicitly.

---

## 4. Proposed Architecture

The new controller should be divided into four layers.

### 4.1 Intent Layer

This is the interface that a human operator, high-level planner, or scripted behavior system controls.

Suggested intent inputs:
- desired travel heading
- desired forward speed
- desired turn rate
- stop / settle request
- style controls:
  - calm <-> tense
  - playful <-> heavy
  - stable <-> chaotic
  - timid <-> determined

This layer should be low-dimensional and human-readable.

### 4.2 Rolling Engine Layer

This is the core authored locomotion system.

It should continue using the spherical-harmonic shape basis, but not as an unconstrained action field. Instead, SH coefficients should be generated by a **structured, phase-conditioned rolling engine**.

The engine should expose interpretable parameters such as:
- rear push amplitude
- front reach amplitude
- support width
- support bias
- duty cycle
- phase speed
- steering bias
- recovery bias
- body settle / anti-puck bias
- expression wobble amount

This turns the current �oracle� into a more explicit locomotion instrument.

### 4.3 Residual Stabilizer Layer

A learned policy sits on top of the rolling engine and outputs bounded corrections.

Its job is:
- heading correction
- phase timing correction
- slip correction
- support-transfer smoothing
- disturbance recovery
- terrain/friction compensation
- avoidance of collapse / puck attractors

This residual should be:
- low amplitude,
- saturating,
- ideally regularized toward zero,
- disabled cleanly for debugging.

The residual should not replace the rolling engine.

### 4.4 Motion Palette / Style Layer

Instead of one �best locomotion policy,� define a family of motion styles.

Examples:
- calm
- determined
- playful
- prowling
- drunk
- stubborn
- injured
- excited

These styles should mostly be implemented by changing rolling-engine parameters and residual limits, not by fully independent policy architectures.

## 5. Why This Is Better Than Broad PPO Refinement

The current repo history strongly suggests a recurring pattern:

- the analytic contact-dipole prior can produce useful rolling,
- BC warmstarts can preserve it,
- PPO can easily damage it when task distribution or reward geometry changes.

That pattern is not a minor training bug. It is a strong signal about what the system wants to be.

For this robot, fully learned locomotion is the wrong default abstraction.

A structured rolling engine plus bounded learning is better because it:
- preserves the discovered locomotion mechanism,
- gives direct control handles,
- supports style variation,
- makes debugging tractable,
- reduces collapse risk,
- aligns with the project�s art/character goals.

## 6. Key Control Rework

### 6.1 Replace �policy outputs locomotion� with �policy modulates locomotion�

The action source should become:

`intent -> rolling engine -> residual correction -> panel targets`

instead of:

`observation -> PPO policy -> panel targets`

or even:

`observation -> residual-on-oracle PPO -> panel targets`

where the oracle is still too monolithic and implicit.

### 6.2 Turn the current oracle into a parameterized generator

The current locomotion prior should be rewritten conceptually as a generator with:
- explicit rolling phase,
- explicit support transfer,
- explicit steering asymmetry,
- explicit amplitude/duty controls,
- optional expressive oscillations.

This should be considered the single most important implementation change.

### 6.3 Make phase explicit

The current locomotion is cyclic. Steering and recovery are phase-dependent.

The system should explicitly track and expose:
- rolling phase
- phase velocity
- support centroid
- contact centroid
- slip estimate
- yaw error relative to desired motion
- body tilt relative to desired motion
- support-transfer confidence

These features should be available to both the rolling engine and the residual policy.

## 7. Recommended Observation Model

The observation stack should emphasize locomotion state, not generic RL completeness.

### 7.1 Required features
- projected gravity / IMU orientation signals
- angular velocity
- linear velocity
- panel extension states
- panel velocity states
- previous rolling-engine outputs
- previous residual outputs
- desired heading / speed / turn rate
- body-frame goal or intent direction
- explicit rolling phase and phase velocity
- contact distribution summary
- support centroid summary
- slip indicators

### 7.2 Contact / tactile features
The existing tactile/contact-field design direction is important for this morphology.

Strong recommendation:
- treat contact distribution across the shell as a first-class sensing modality,
- derive low-dimensional summaries suitable for closed-loop control,
- preserve the option to later build more expressive tactile perception.

For locomotion specifically, the most important contact-derived quantities are likely:
- which region is currently supporting load,
- whether the support patch is where the rolling engine expects it to be,
- whether support transfer is early/late,
- whether the robot is slipping instead of rolling.

### 7.3 Features to avoid over-emphasizing
Do not bloat the controller with observations that make sense for generic benchmark RL but are not central to this locomotion mode.

The observation design should stay focused on:
- phase,
- contact,
- tilt,
- heading,
- support transfer,
- controlled asymmetry.

## 8. Recommended Training Pipeline

The reworked training pipeline has four stages.

### 8.1 Stage A: Primitive Authoring

Create a library of scripted / parameterized rolling primitives using the new rolling engine.

This is not one oracle. It is a family.

Minimum primitive set:
- straight roll
- gentle left arc
- gentle right arc
- accelerate
- decelerate / settle
- hold heading while rolling
- recover from off-axis start
- recover from poor support transfer
- expressive wobble without large displacement
- stop without collapsing into puck-like failure

Outputs from this stage:
- a stable rolling engine API,
- artist-facing parameters,
- test scenarios for each primitive,
- a primitive dataset recorder.

### 8.2 Stage B: Imitation Learning

Train a policy to imitate the primitive family across randomized conditions.

Randomize:
- heading
- starting yaw
- speed command
- turn command
- friction
- minor body perturbations
- initial support configuration
- style parameters

The purpose of this stage is:
- compress the motion family into a policy,
- make interpolation between primitives easier,
- create a smooth controller manifold,
- preserve authored behavior.

Success criterion:
the cloned policy should feel like a faithful compact version of the primitive library, not a reward-optimized deviation from it.

### 8.3 Stage C: Corrective Imitation / DAgger-Style Expansion

After initial cloning, the controller will drift into states the scripted library did not cover well.

Use an iterative corrective loop:
1. run the learned controller,
2. detect drift/failure regions,
3. query scripted recovery logic or hand-authored correction logic,
4. add those states back into the dataset,
5. retrain.

This stage is likely more important than PPO for this robot.

Focus correction collection on:
- bad yaw starts
- failed support-transfer timing
- slip-heavy states
- partial puck attractor states
- perturbation recovery
- transitions between motion styles

### 8.4 Stage D: Narrow Residual RL

Only after imitation + corrective data are working should RL be applied.

RL should optimize a small bounded residual over the rolling engine / imitation controller.

RL target areas:
- friction robustness
- phase timing refinement
- heading correction under disturbance
- stability during aggressive steering
- recovery after impulsive error
- robustness to terrain micro-variation

RL should **not** be allowed to redefine the base gait.

Recommended constraints:
- residual magnitude limits
- action regularization toward zero residual
- KL / anchor regularization toward the imitation policy
- frequent visual review
- explicit regression checks against authored controllability

## 9. Optional Search Methods

### 9.1 CMA-ES or related evolutionary search
If a search method is used, it should optimize the **rolling engine parameters** or primitive generator parameters, not a full large neural policy.

Good search targets:
- phase offsets
- duty cycle schedules
- rear/front amplitude ratios
- steering asymmetry gains
- recovery biases
- style-specific primitive presets

Bad search target:
- the entire end-to-end locomotion policy.

Reason:
the generator is low-dimensional, structured, and interpretable. That is where gradient-free search is likely useful.

### 9.2 Symmetry use
Geometric symmetry can still be exploited where appropriate, but it should support the rolling engine and training data, not drive the architecture.

## 10. Character-Oriented Motion Design

Because this robot is also a character experiment, motion style should be designed deliberately.

### 10.1 Recommended style axes
- calm <-> agitated
- heavy <-> light
- deliberate <-> twitchy
- confident <-> hesitant
- stable <-> chaotic
- curious <-> stubborn

### 10.2 How style should manifest
Style should affect:
- phase speed
- support width
- contact aggressiveness
- steering overshoot
- amount of body wobble
- acceleration profiles
- recovery urgency
- stop behavior
- settle posture

### 10.3 Example motion palette
- **Calm:** low phase speed, wide support, low overshoot, stable turns
- **Playful:** faster phase, mild bounce, larger steering overshoot, expressive wobble
- **Heavy:** slower acceleration, stronger ground commitment, low chatter
- **Drunk:** delayed correction, broader lateral sway, inconsistent support transfer
- **Determined:** strong heading hold, aggressive recovery, reduced expressive wobble
- **Injured:** asymmetric amplitudes, slower recovery, shortened stride effect

The long-term goal is for the robot to feel like a controllable performer, not just a solved locomotion benchmark.

## 11. Evaluation Philosophy

The evaluation stack should shift from �single scalar reward proves progress� to �controllability and motion quality gates progress.�

### 11.1 Primary evaluation categories
- can start reliably
- can maintain rolling rhythm
- can hold heading
- can arc left and right on command
- can slow down and stop without collapse
- can recover from poor initial yaw
- can recover from minor disturbances
- can preserve distinct styles while remaining controllable

### 11.2 Regression suite
Every major stage should render and review:
- straight-line starts
- left/right arcs
- aggressive turn initiation
- stop/settle
- disturbed mid-roll recovery
- off-axis spawn recovery
- style-switch transitions

### 11.3 Success criteria
A training stage is successful only if it improves capability **without erasing authored controllability**.

For this project, �more reward, less character� is not a win.

## 12. Implementation Roadmap

### Phase 1: Formalize the Rolling Engine
Goal:
turn the current contact-dipole locomotion logic into a parameterized, phase-aware rolling engine.

Tasks:
- define engine parameters
- define explicit phase state
- define support-transfer logic
- define SH coefficient synthesis from engine state
- expose steering and style controls
- build debug visualizations for phase and support regions

Deliverable:
a deterministic controllable rolling engine that works without learning.

### Phase 2: Primitive Library
Goal:
author a motion palette of locomotion primitives.

Tasks:
- straight roll
- arc left / arc right
- accelerate / brake
- recovery primitives
- stop / settle primitive
- expressive idle / wobble primitive
- style presets

Deliverable:
a library of callable primitive generators with recording support.

### Phase 3: Dataset and Imitation Stack
Goal:
record a broad family of demonstrations and train a compact policy to imitate them.

Tasks:
- dataset format for primitive family
- randomized recording sweeps
- behavior cloning training loop
- evaluation against primitive fidelity
- visual review and failure catalog

Deliverable:
an imitation controller that reproduces the primitive family.

### Phase 4: Corrective Data Loop
Goal:
close the imitation gap in off-distribution states.

Tasks:
- failure detector instrumentation
- targeted drift scenarios
- scripted corrective supervisor queries
- DAgger-style aggregation
- retraining iterations

Deliverable:
a much more stable controllable policy without broad RL drift.

### Phase 5: Residual RL
Goal:
add bounded robustness improvements while preserving authored behavior.

Tasks:
- residual action head or residual policy wrapper
- zero-centered regularization
- anchor / KL preservation against imitation controller
- disturbance and friction curricula
- visual approval gates

Deliverable:
a robust controller that still feels authored.

### Phase 6: Character Layer
Goal:
turn the locomotion controller into a style-capable character controller.

Tasks:
- style parameter interface
- named style presets
- transition rules between styles
- operator-facing command mapping
- presentation / demo scenarios

Deliverable:
a controllable expressive rolling character system.

## 13. Immediate Next Step

The first implementation target should be:

**Re-express the current scripted locomotion as a phase-conditioned rolling engine with explicit parameters and explicit phase/support state.**

Not PPO tuning.  
Not reward redesign.  
Not terrain generalization.

That is the highest-leverage move because all later stages depend on having a controllable substrate rather than an opaque oracle.

Immediate sub-goals:
1. define the rolling engine parameter list,
2. define the explicit phase state,
3. map engine state to SH coefficients,
4. validate straight rolling and simple steering without learning,
5. build dataset recording around that engine.

---

## 14. Non-Goals

This rework is **not** trying to:
- replace the spherical-harmonic locomotion idea,
- train end-to-end from scratch,
- pursue generic RL benchmark performance,
- maximize reward at the expense of legibility,
- collapse all behavior into one monolithic policy,
- assume the robot must be purely practical.

This rework is trying to make the robot:
- controllable,
- expressive,
- robust enough,
- characterful,
- structurally understandable.

---

## 15. Guidance for Future Implementation Sessions

When implementing this plan, prefer these defaults:

- preserve the contact-dipole rolling prior,
- preserve SH-based low-dimensional shape control,
- introduce explicit rolling phase,
- keep the learned component bounded,
- prefer imitation and corrective data over unconstrained PPO,
- evaluate by rendered motion quality and controllability,
- treat style variation as a first-class requirement,
- refuse changes that make the robot �better optimized� but less interpretable.

If a future session faces a choice between:
- a more elegant end-to-end learning setup, and
- a more controllable authored rolling engine,

choose the authored rolling engine.

That choice is the point of this rework.
